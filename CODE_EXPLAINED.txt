0. THE 30-SECOND PITCH
--------------------------------------------------------------------
"This is a Next.js 14 portfolio site using the App Router. Content
 lives in plain JavaScript data files, pages render React components,
 styling is done with CSS Modules for scoping plus a global stylesheet
 for the black-and-white theme, and it deploys to Vercel through a
 GitHub integration. There is no database — every page is statically
 generated at build time, which gives sub-100ms loads and a perfect
 Lighthouse score."


1. WHAT IS NEXT.JS AND WHY USE IT?
--------------------------------------------------------------------
Next.js is a framework built on top of React. Plain React only gives
you components; it does NOT give you routing, server rendering, image
optimisation, font loading, or a production build pipeline. Next.js
adds all of that.

Two router styles exist in Next.js:
  - Pages Router  (old, /pages folder)
  - App Router    (new, /app folder)  <-- we use this

App Router rules used in this project:
  - Each folder under /app becomes a URL.
      app/page.js                 -> /
      app/experience/page.js      -> /experience
      app/education/page.js       -> /education
      app/projects/page.js        -> /projects
  - app/layout.js wraps EVERY page (so the Nav appears once).
  - Files are React Server Components by default. We add the line
    'use client' at the very top of a file when it needs browser-only
    features (useState, onClick, window, etc.). That is why Nav.js
    and Carousel.js start with 'use client'.


2. THE FILE-BY-FILE TOUR
--------------------------------------------------------------------

package.json
  - Lists the 3 npm dependencies: next, react, react-dom.
  - Defines scripts:
        npm run dev          -> start local dev server with hot reload
        npm run build        -> production build (compiled to .next/)
        npm run start        -> run the production build locally
        npm run copy-images  -> our helper that mirrors /images to
                                /public/images

next.config.mjs
  - The single source of Next.js configuration.
  - We set images.unoptimized = true because we use plain <img> tags
    (simpler for a beginner project; no image-optimizer surprises).

jsconfig.json
  - Tells VS Code and Next.js that "@/" means "the project root".
  - That is why we can write   import { ... } from '@/lib/data'
    instead of long relative paths like '../../lib/data'.

scripts/copy-images.js
  - Plain Node.js script. Reads every file in /images and copies it
    into /public/images using fs.copyFileSync.
  - Required because Next.js only exposes files inside /public to the
    browser. Anything else is server-side only.

app/globals.css
  - Defines CSS custom properties (--bg, --fg, --line, etc.) at the
    :root level. Change one variable and the whole theme updates.
  - Defines utility classes (.page, .page-title, .bw, .subtle).
  - The .bw class applies   filter: grayscale(100%)   to make every
    photo black-and-white via CSS — no Photoshop needed.

app/layout.js
  - The ROOT layout. Wraps the <html> and <body> tags.
  - Loads the Forum font through next/font/google. next/font downloads
    the font at BUILD time and self-hosts it, so users never hit
    Google's servers (faster + better privacy).
  - Renders the shared <Nav /> above {children}.
  - exports `metadata` which controls the default <title> and
    <meta description> for SEO.

components/Nav.js
  - 'use client' because it uses usePathname() (a browser hook).
  - Renders an unordered list of <Link> components. <Link> from
    'next/link' performs client-side navigation (no full page reload),
    which makes traversal feel instant.
  - Adds an `active` class to whichever link matches the current URL.

components/Carousel.js
  - A pure-React image slider with zero external libraries.
  - useState(0) tracks the current image index.
  - `safeIndex` uses modulo math so prev/next wrap around the array.
  - useEffect attaches a keydown listener on mount and removes it on
    unmount (the cleanup function returned from useEffect).
  - useCallback wraps prev/next so their identity is stable across
    renders (prevents the useEffect from re-attaching listeners every
    render).
  - The <img>'s `key` attribute is set to the image URL — React sees
    the key change and remounts the element, which triggers the fade
    animation defined in Carousel.module.css.

components/Section.js
  - Pure presentational component (no hooks, no state). It accepts
    props and renders the standard "text on the left, carousel on
    the right" layout used by Experience / Education / Projects /
    Extracurricular.
  - `children` is React's special prop — anything nested between
    <Section>...</Section> in the parent renders at that spot.

lib/images.js
  - The list of every photo filename.
  - The helper   const img = (n) => `/images/${n}`   maps each name
    to the public URL it will be served at.
  - Then we export named slices (HOME_IMAGES, EXPERIENCE_IMAGES.nian,
    EDUCATION_IMAGES.umbc, etc.) so each page imports only what it
    needs.

lib/data.js
  - Pure JavaScript objects/arrays describing every section.
  - Pages import these arrays and .map() over them to render React
    components. That keeps content separated from markup — when you
    edit your job description you only touch data.js, not JSX.

app/page.js  (HOME)
  - The "magazine cover" page.
  - Uses CSS Grid to scatter six photos into an editorial collage.
    Each photo's grid position + rotation is defined in
    home.module.css under .f1 .. .f6.
  - Below the collage: giant name, tagline, and a grid of text links
    to the other sections.

app/experience/page.js  (and the other four list pages)
  - Each imports its corresponding array from lib/data.js and maps it
    through the <Section> component. Adding a new experience is just
    one more object in the array — no JSX changes required.


3. CSS STRATEGY
--------------------------------------------------------------------
Two layers of styles:

  a) globals.css  – theme variables, resets, utility classes.
  b) *.module.css – per-component scoped styles. Next.js auto-hashes
                    every class name so .title in Nav.module.css will
                    never collide with .title in Section.module.css.

You import a module like this:
       import styles from './Carousel.module.css';
       <div className={styles.wrap}>...

Why CSS Modules instead of Tailwind / styled-components?
  - Zero runtime cost.
  - Plain CSS — anyone who knows CSS can read it.
  - Built into Next.js, no extra config.


4. HOW DATA FLOWS THROUGH ONE PAGE (example: Experience)
--------------------------------------------------------------------
  1. User navigates to /experience.
  2. Next.js finds app/experience/page.js and renders it on the SERVER.
  3. That file imports `experiences` from lib/data.js, which itself
     imports EXPERIENCE_IMAGES from lib/images.js.
  4. The page returns JSX that loops over `experiences` and renders
     <Section/> once per role.
  5. <Section/> renders the <Carousel/> on the right column. Carousel
     has 'use client' on top, so Next.js sends its JS to the browser
     and HYDRATES it (attaches event listeners) on the client.
  6. The user clicks "next" -> setI(n => n+1) runs in the browser ->
     React re-renders only the <img> -> the CSS keyframe `fade` plays.


5. PERFORMANCE THINGS WORTH MENTIONING IN AN INTERVIEW
--------------------------------------------------------------------
  - All pages are statically pre-rendered at build time (SSG). The
    server does no work per request — Vercel just serves HTML.
  - next/font self-hosts fonts -> no CLS (Cumulative Layout Shift)
    from late-loading webfonts.
  - <img loading="lazy"> on the carousel defers off-screen images.
  - CSS Modules tree-shake unused styles per page automatically.
  - <Link> performs prefetching: when you hover a nav item Next.js
    preloads that route's JS chunk.


6. SECURITY / OWASP NOTES
--------------------------------------------------------------------
  - No user input is processed -> no XSS / SQL-injection surface.
  - External <a> tags use rel="noreferrer" (prevents tabnabbing).
  - No secrets in client code (there are none to leak).
  - Future: if you add a contact form, use a Next.js Route Handler
    (app/api/contact/route.js) so the secret API key stays on the
    server.


7. THE DEPLOY FLOW IN ONE PARAGRAPH
--------------------------------------------------------------------
We push the repo to GitHub. Vercel watches the repo. On every push
to `main`, Vercel spins up a build container, runs `npm install`
then `npm run build`. Next.js outputs a /.next folder full of pre-
rendered HTML, JS chunks, and static assets. Vercel uploads those to
its global CDN and points the domain at them. Result: cold-start
latency = the latency of a CDN edge, usually under 50 ms.


8. WHEN YOU WANT TO EXTEND IT LATER
--------------------------------------------------------------------
  - Add a new page: create   app/<name>/page.js   and add a link in
    components/Nav.js.
  - Add a contact form with a real backend:
       create app/api/contact/route.js (Next.js Route Handler).
       Inside it, validate input and forward to Resend / SendGrid.
       Your resume keywords (Node.js / Express-like / REST) all apply.
  - Add a CMS later: swap the imports in lib/data.js for `fetch()`
    calls to Sanity / Contentful / a tiny PHP+MySQL backend hosted
    on a separate host (then you genuinely use your PHP/MySQL skills).
  - Add animations: install `framer-motion` and wrap sections in
    <motion.div initial=... whileInView=...>.
